In the IDS platform, there exists at least one connector per IDS customer.
Each customer can be a Data Provider or a Data Consumer.
There are variety of Data Usage Control policies that can be defined to protect the customers data.
In this tool, we have implemented an editor that supports the customers to specify their Data Usage Control policies.

As an IDS customer, you can choose a policy type from the side bar and then, follow the instructions.
You will get the corresponding ODRL policy.
In addition, we have implemented a mapping tool that translates an ODRL policy to a MYDATA policy. This policy can be deployed to MYDATA framework to technically protect your data.

Actions

ids:read
Similar concept to odrl:read. When there is no interface defined, it means http get.
It may get ids:system, ids:purpose, ids:encoding or ids:count as a left operand.
ids:read a odrl:Action

ids:delete
Similar concept to odrl:delete. In order to remove all copies of an artifact in the consumer connector (assumption is that there is no copy of that data outside of the consumer connector)
It gets ids:delay period as a left operand.

ids:log
In order to log information to a known logging server.
It gets ids:recipient as a left operand.

ids:inform
Similar concept to odrl:inform. In order to notify an ids:informedParty that an action has been performed.

ids:print
Similar concept to odrl:print.

ids:reproduce
Similar concept to odrl:reproduce. In order tn order to make a duplicate copy to the consumer side connector.

ids:display
Similar concept to odrl:display. In order to visually render a resource (artifact).

ids:distribute
Similar concept to odrl:distribute. In order to send data out of the connector (unauthorized party)


ids:anonymize
Similar concept to odrl:anonymize. This is an abstract action.

ids:round
This class extends ids:anonymize action. It gets an ids:jsonPath and an ids:digit as input parameters and returns a rounded number value.
