Data Usage Control Scenario

An IDS customer can be a Data Provider or a Data Consumer.
There are variety of Data Usage Control policies that can be defined by data providers to protect their data.
The Open Digital Rights Language (ODRL) is a policy expression language that provides a flexible and interoperable information model, vocabulary, and encoding mechanisms for representing statements about the usage of content and services.
ODRL policies are technology-independent and are not made to be technically enforced to a system. Therefore, it is been decided to use this language as a baseline for other technology-dependent solutions that can enforce Data Usage Control policies into the systems to protect data.
One of the technology-dependent policy languages and platforms provided by Fraunhofer IESE is MYDATA.
MYDATA is a solution that intercepts events or data flows in a system and enforces a security decision based on MYDATA policies.

PAP Editor
In this tool, we have implemented an editor that supports the data providers to specify their Data Usage Control ODRL policies.
In addition, they can use this tool to translate the ODRL policies to MYDATA policies.

How To
As an IDS Data Provider, you can choose a policy class from the side bar of the tool and then, follow the instructions.
By pressing the "Save" button, you will get the corresponding ODRL policy.
In order to get the MYDATA policy, you can press the "Transfer" button.
In addition, you can export the MYDATA policy from this editor.
This exported MYDATA policy is ready to be imported into the MYDATA PMP (Policy Management Point).
You can then deploy the MYDATA policy.
Deploying MYDATA policy into the MYDATA PDP (Policy Decision Point) is not enough for the policy enforcement.
Be aware that you still need to create MYDATA PEP (Policy Enforcement Point) and the needed MYDATA PIPs (Policy Information Points) and MYDATA PXPs (Policy Execution Points).
When the enforcement point is implemented, then you can be assured that your data is technically protected. To implement your MYDATA PEP (Policy Enforcement Point), follow the instructions documented in MYDATA Web Page.

Actions

ids:read
Similar concept to odrl:read. When there is no interface defined, it means http get.
It may get ids:system, ids:purpose, ids:encoding or ids:count as a left operand.
ids:read a odrl:Action

ids:delete
Similar concept to odrl:delete. In order to remove all copies of an artifact in the consumer connector (assumption is that there is no copy of that data outside of the consumer connector)
It gets ids:delay period as a left operand.

ids:log
In order to log information to a known logging server.
It gets ids:recipient as a left operand.

ids:inform
Similar concept to odrl:inform. In order to notify an ids:informedParty that an action has been performed.

ids:print
Similar concept to odrl:print.

ids:reproduce
Similar concept to odrl:reproduce. In order tn order to make a duplicate copy to the consumer side connector.

ids:display
Similar concept to odrl:display. In order to visually render a resource (artifact).

ids:distribute
Similar concept to odrl:distribute. In order to send data out of the connector (unauthorized party)


ids:anonymize
Similar concept to odrl:anonymize. This is an abstract action.

ids:round
This class extends ids:anonymize action. It gets an ids:jsonPath and an ids:digit as input parameters and returns a rounded number value.
